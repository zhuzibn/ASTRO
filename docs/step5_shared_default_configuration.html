Step 5 - Shared Default Configuration

Current Status

The working tree currently contains Step 4 changes for rejecting unsupported solver modes. Step 5 has not been implemented yet in the current diff. This report explains the intended Step 5 design and verification path rather than documenting completed source edits.

Summary

Problem

main.m and benchmark/run_baseline_benchmark.m both define the production physics and time-control defaults. The benchmark comments mark those values as copied from main.m, which creates drift risk.

Goal

Create one explicit default-configuration source, then make production and benchmark paths consume the same values while keeping benchmark-only overrides documented and narrow.

Expected Benefit

Default physics, solver selection, enabled terms, and time-grid settings change in one place. The regression benchmark remains aligned with production defaults unless it deliberately overrides lattice size or deterministic input data.

Main Risk

ASTRO scripts depend on workspace variables. A missing variable, renamed field, or changed assignment order can silently alter numerical behavior or fail only after GPU integration begins.

Design Intent

Step 5 should add a small, explicit configuration function such as astro_default_config.m. Its job is to centralize default scalar values, vectors, flags, physical parameters, and time settings currently duplicated between production and benchmark execution.

The function should not become a general framework. It should preserve current values exactly and expose only the data needed by existing scripts. Because the current solver path is workspace-oriented, the safest implementation is likely a simple struct plus explicit assignment at each call site.

- %% Control parameters copied from main.m- rk4 = 1;- bc = 1;- DMIenable = 1;- gpusave = 1e-12;- tstep = 2e-16;+ cfg = astro_default_config();+ % Explicitly assign cfg fields needed by legacy workspace scripts.+ rk4 = cfg.rk4;+ bc = cfg.bc;+ DMIenable = cfg.DMIenable;+ gpusave = cfg.gpusave;+ tstep = cfg.tstep;

Per-file Change Plan

astro_default_config.m

Add a new default source that returns the production configuration. Include solver flags, boundary settings, enabled field terms, material constants derived from constantfile.m, torque settings, thermal settings, lattice defaults, and time-control defaults. Keep units visible in comments where values represent physical quantities.

main.m

Replace local default blocks with consumption of astro_default_config(). Keep production behavior unchanged: random atom distribution when no fixed distribution is requested, natomW=20, natomL=30, compositionn=0.1, d=0.4e-9, rk4=1, and the same time grid.

benchmark/run_baseline_benchmark.m

Consume the same defaults, then apply benchmark-specific overrides only for documented regression needs: quick/current lattice size, fixed atom distribution file, deterministic seed/input handling, and candidate output paths. The parameter snapshot should still record the effective values used by the run.

tests/run_smoke_tests.m

Add or preserve smoke checks that prove the benchmark output still matches the intended configuration: trajectory shape, saved-time contract, finite values, spin normalization, atom distribution metadata, and unsupported solver rejection.

README.md, AGENTS.md, CHANGELOG.md

Document the shared configuration source and update the changelog only when the source implementation is actually made. Do not update benchmark baseline artifacts unless explicitly approved.

Behavioral Contract

  • main.m and the benchmark must use identical production defaults before benchmark overrides are applied.
  • Step 5 must not change physics formulas, signs, random-distribution semantics, solver choice, time indexing, spin normalization, or accepted baseline artifacts.
  • rk4=1 remains the only supported production solver unless a later task explicitly validates another solver path.
  • The benchmark may override lattice size and atom distribution for reproducibility; those overrides must remain obvious in code and documentation.
  • Any comparison should use the accepted post-time-contract candidate, not an older incompatible trajectory.
