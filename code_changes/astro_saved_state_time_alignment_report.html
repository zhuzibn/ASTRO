ASTRO Saved-State/Time Alignment Change Report

Current branch changes for aligning saved spin states with saved timestamps.

Scope: report only. No MATLAB source code is modified by this document.

Report file: code_changes/astro_saved_state_time_alignment_report.html

1. Summary

The change defines saved spin states and saved times as the same physical samples. The trajectory now includes the initial state at t=0, includes the final state at runtime, and avoids duplicate states at GPU chunk boundaries.

The implementation is limited to saved-time construction and RK4 trajectory assembly. It does not change field formulas, torque formulas, boundary semantics, normalization, or atom-distribution behavior.

Question: Explain these lines. If gpurun_number=3, do they also give the correct result?

if ct3==1
    saveindices=1:savetstep:(gpusteps+1);
else
    saveindices=(savetstep+1):savetstep:(gpusteps+1);
end

Yes. The logic still gives the correct result for gpurun_number=3, assuming gpusteps is an integer multiple of savetstep, which the code already enforces.

These are local indices inside one GPU chunk. Each chunk now has gpusteps+1 stored states:

mmx(:,:,1)              % chunk initial state
mmx(:,:,2)              % after 1 tstep
...
mmx(:,:,gpusteps+1)     % after gpusteps tsteps, chunk endpoint

For the first chunk, saveindices=1:savetstep:(gpusteps+1) saves the initial state and every savetstep update after it:

1, 101, 201, ..., 5001
t = 0, 2e-14, 4e-14, ..., 1e-12

For later chunks, saveindices=(savetstep+1):savetstep:(gpusteps+1) starts at local index 101, not 1, because local index 1 is the carried state from the previous chunk endpoint. Saving it again would duplicate the chunk boundary.

For gpurun_number=3, the saved chunks are:

chunk 1: 1, 101, 201, ..., 5001
         t = 0 ... 1e-12

chunk 2: 101, 201, ..., 5001
         t = 1.02e-12 ... 2e-12

chunk 3: 101, 201, ..., 5001
         t = 2.02e-12 ... 3e-12

The total saved count is:

1 + gpurun_number * (gpusteps / savetstep)
1 + 3 * (5000 / 100) = 151

That matches the saved-time vector:

t = (0:savetstep:totstep)*tstep
totstep = 3 * 5000 = 15000
numel(0:100:15000) = 151

Therefore the logic remains correct for gpurun_number=3: the first state is t=0, the last state is runtime, times are strictly increasing, and chunk boundaries are not duplicated.

4. Behavioral Contract

  • The first saved state is the initialized spin state at t=0.
  • The last saved state is the spin state at runtime.
  • numel(t) equals size(mmx,3), size(mmy,3), and size(mmz,3).
  • Saved times are strictly increasing.
  • Chunk boundary states are not duplicated.

For gpurun_number=3, the saved count is 1 + 3*(gpusteps/savetstep). With gpusteps=5000 and savetstep=100, this gives 151 saved states from 0 through 3e-12.
